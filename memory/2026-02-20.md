## 01:22 - CCCC Architect Actor 修复

### 问题描述
8848 WebUI 显示 Claude Code (architect/foreman) 已停止，而 Codex (coder) 显示运行正常。

### 根因分析
- `architect` actor 的命令是 `claude --dangerously-skip-permissions`
- Claude Code 在 root/sudo 环境下禁止使用 `--dangerously-skip-permissions` 参数
- 启动时直接报错退出，但 daemon 仍认为 actor 是 "on" 状态（没有 stop 事件）

### 解决方案
```bash
cccc actor update architect --group g_c9581a21c73a --command "claude --permission-mode acceptEdits"
cccc actor restart architect --group g_c9581a21c73a
```

修改后的命令在 root 环境下可以正常运行。

### 配置位置
`~/.cccc/groups/g_c9581a21c73a/group.yaml` 第 20-22 行

## 01:10 - Telegram 报错通知问题

### 问题描述
用户反馈：每次对话最后一条消息会被更新成报错信息，例如：
```
⚠️ 🛠️ Exec: bash -lc 'cd ~/.openclaw/workspace/cccc && . .venv/bin/activate && python3 - <<"PY" failed: Traceback (most recent call last):
```

### 根因分析
- OpenClaw 会自动发送 exec 失败通知到 Telegram
- 当 `bash -lc 'cmd1; cmd2; cmd3'` 中任意命令返回非零退出码时，整体标记为 failed
- 常见触发场景：
  1. `grep` 没匹配到结果（返回 exit code 1）
  2. Python 脚本报错
  3. 命令语法错误

### 解决方案
执行检查命令时使用"永不报错模板"：
```bash
bash -lc 'pgrep -a claude || echo "no claude"; pgrep -a codex | head -3 || echo "no codex"; ss -ltnp | grep 8848 || echo "8848 down"; echo "OK"'
```

关键点：
- 每个可能失败的命令后加 `|| echo "fallback"`
- 最后加 `echo "OK"` 确保整体返回成功

## 16:25 - Telegram 群组回复修复

### 问题描述
在 Telegram 群组 `4882522885` 中 @机器人 没有回复。

### 根因
- `groupPolicy: "allowlist"` 模式下，需要同时配置群组允许列表和发送者允许列表
- 之前只配置了群组 `enabled: true`，没有配置 `groupAllowFrom`

### 解决方案
通过 `gateway config.patch` 添加：
```json
{
  "channels": {
    "telegram": {
      "groups": {
        "4882522885": {
          "enabled": true,
          "requireMention": true
        }
      },
      "groupAllowFrom": ["6472343782"],
      "groupPolicy": "allowlist"
    }
  }
}
```

### 消息反应配置恢复
发现 `messages.ackReactionScope` 丢失，重新添加：
```json
{
  "messages": {
    "ackReaction": "👀",
    "ackReactionScope": "group-mentions",
    "removeAckAfterReply": true
  }
}
```
